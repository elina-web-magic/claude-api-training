=== PROMPT 1: GOALS & ARCHITECTURE ===
# AILens System Architecture Overview

## 1. Main Goals of the System

- **Research Discovery**: Enable users to search, aggregate, and filter AI research papers/articles across multiple sources
- **Intelligent Curation**: Use AI to surface relevant research based on user interests and reading patterns
- **Collaboration**: Allow researchers to share findings, annotations, and insights within teams
- **Multi-language Support**: Serve a global research community with localized content
- **Real-time Updates**: Continuously ingest and process new research publications via automated jobs

## 2. High-Level Architecture

**Frontend Layer (Next.js App Router)**
- Server-side rendered pages for SEO and performance
- Dynamic routes for research papers, user profiles, and team workspaces
- API routes as lightweight middleware for client-side operations
- i18n middleware for language detection and routing

**Backend Services**
- **Prisma ORM**: Data modeling and migrations for user profiles, research collections, annotations, and team management
- **Vercel Postgres**: Primary data store (normalized schema for relational integrity)
- **Cron Jobs**: Scheduled tasks for content ingestion, AI processing (embeddings/tagging), and digest generation
- **External APIs**: Integration with arXiv, PubMed, or Semantic Scholar for research feeds

**Data Processing Pipeline**
- Cron-triggered workers process new papers asynchronously
- Generate embeddings for vector-based similarity searches
- Categorize and tag research using AI models
- Send notifications to users with curated recommendations

**Deployment**
- Vercel hosting (Next.js optimized, auto-scaling)
- Serverless functions for API endpoints and background jobs

## 3. Pros and Cons

### Pros
- **Fast TTM**: Next.js + Vercel eliminates infrastructure management; focus on features
- **Type Safety**: Prisma + TypeScript reduce runtime errors and improve code maintainability
- **Scalability**: Vercel's serverless architecture handles traffic spikes; Postgres scales vertically
- **i18n Built-in**: Native multi-language support without additional frameworks
- **DX**: App Router simplifies file-based routing and co-location of features
- **Cron Reliability**: Managed cron jobs reduce operational overhead

### Cons
- **Database Limitations**: Vercel Postgres (managed) has connection pool limits; may bottleneck with high concurrency
- **Serverless Cold Starts**: Background jobs may face latency; not ideal for real-time processing
- **Cost Scaling**: Vercel + Postgres pricing escalates quickly with data growth or high request volume
- **Limited Customization**: Managed services reduce control over optimization and infrastructure decisions
- **Cron Job Constraints**: Limited scheduling flexibility; third-party integrations (external APIs) require careful timeout handling
- **Vector Search Gap**: Vercel Postgres lacks native vector search; requires pgvector extension setup and maintenance complexity

## Recommendation for MVP

This stack is **ideal for MVP validation** due to low operational overhead and rapid deployment. However, plan migration paths:
- **Phase 2**: Consider dedicated vector database (Pinecone, Weaviate) if similarity search becomes a core feature
- **Phase 3**: Evaluate dedicated job queue (Bull, RabbitMQ) if background processing scales significantly

Focus initial efforts on core research discovery and user onboarding rather than premature optimization.

## 2. Cron Job: arXiv RSS Feed Ingestion Flow

### Phase 1: Initialization & Validation
- **Validation**: Verify arXiv RSS endpoint availability via health check (timeout: 5s)
- **Logging**: Create CronJobLog entry with `pending` status
- **Lock**: Implement distributed lock (Redis/database) to prevent concurrent executions

### Phase 2: Data Fetching
- **Fetch**: Retrieve latest feed (configurable: last 24 hours or last 100 articles)
- **Parse**: Extract `arxivId`, `title`, `authors`, `abstract`, `publishedAt`, `category`, `pdfUrl`
- **Validation**: Validate required fields; reject malformed entries
- **Deduplication**: Check `arxivId` against existing Article records

### Phase 3: Granular Error Handling
- **Transient Errors** (network timeouts, temporary 5xx): Retry with exponential backoff (3 attempts, base 2s)
- **Parsing Errors**: Log skipped articles per entry; continue processing remainder
- **Constraint Violations** (duplicate arxivId): Log as warning; skip entry
- **Fatal Errors** (database connection loss): Rollback transaction; mark job as `failure`

### Phase 4: Data Ingestion & Storage
- **Batch Insert**: Insert Article records in bulk (batch size: 50) with ON CONFLICT handling
- **Atomic Transaction**: Wrap all inserts in single transaction for consistency
- **Locale Handling**: Set default `locale` to 'en'; support future multi-language feeds via config

### Phase 5: Post-Processing
- **Update Metadata**: Increment ArticleMetadata.viewCount for indexing
- **Cleanup**: Archive old articles (>90 days) if needed
- **Logging**: Record success/failure counts; update CronJobLog status
- **Alerts**: Trigger notifications if failure rate exceeds threshold (e.g., >20% articles failed)

### Phase 6: Monitoring
- **Metrics**: Track processing duration, success/failure rates, throughput
- **Scheduled Frequency**: Run every 6 hours (configurable)

## 2. Cron Job: arXiv RSS Ingestion Flow

### Execution Trigger
- **Schedule**: Every 6 hours (configurable via environment)
- **Timezone**: UTC

### Logical Flow

**Phase 1: Initialization**
- Generate unique `jobRunId`
- Retrieve last successful `ingestedAt` timestamp from IngestionLog
- Log job start with PENDING status

**Phase 2: Feed Fetch**
- Construct arXiv RSS URL with query filters (e.g., cs.AI, cs.LG categories)
- Execute HTTP GET with 30-second timeout
- **Error Handling**: Network failures → Retry with exponential backoff (3 attempts, 5s/10s/20s delays)
- **Validation**: Verify RSS response format (XML structure) before parsing

**Phase 3: Feed Parsing**
- Parse XML entries into structured objects
- Extract: `arxivId`, `title`, `abstract`, `authors`, `publishedAt`, `categories`, `pdfUrl`
- **Validation**: 
  - arxivId format (regex: `\d{4}\.\d{5}(v\d+)?`)
  - publishedAt ≤ current timestamp
  - Title/abstract non-empty
  - Categories array not empty
- **Granular Error**: Log malformed entries; skip with warning (don't fail entire job)

**Phase 4: Deduplication**
- Query existing articles by `arxivId`
- Separate into: NEW (insert) vs. EXISTING (update if `lastModifiedAt` differs)

**Phase 5: Database Transaction**
- Batch upsert articles (MAX 100 per transaction)
- On constraint violation (duplicate arxivId): Update existing record
- **Transactional Safety**: Rollback on database errors; retry transaction once
- **Error Handling**: Constraint violations → Log and continue (partial success)

**Phase 6: Logging & Cleanup**
- Update IngestionLog with final status
- Count: processed, inserted, updated, failed
- Log all errors to JSONB errorDetails field
- **Determination**: 
  - All success → Status: SUCCESS
  - Some failures → Status: PARTIAL_FAILURE
  - Complete failure → Status: FAILURE (trigger alert via webhook)

### Retry Strategy
- **Network failures**: 3 exponential backoff attempts
- **Database transient errors**: Single retry after 2s delay
- **Validation failures**: Log and skip; continue processing (non-blocking)
- **Max job duration**: 5 minutes hard timeout
